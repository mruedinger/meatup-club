# Lessons Learned

- 2026-02-21: Initialized lessons log for remediation tracking.
- 2026-02-21: Route-action DB mocks must support both `prepare().first/run/all` and `prepare().bind().first/run/all`; otherwise tests can miss real query shapes and fail with mock-only type errors.
- 2026-02-23: When introducing stricter shared DB types, update dependent tests and typed query result casts in the same change; otherwise typecheck breaks on mock shape mismatches and `unknown` row types.
- 2026-02-23: Route-manifest integrity tests should use simple regex literals and validated filename normalization (`/\.tsx?$/`), because over-escaped patterns can silently invalidate the test harness.
- 2026-02-23: Failure mode: treating repo schema as default authority when live DB was the intended source; detection signal: table/constraint drift and user correction to reverse direction; prevention rule: for schema reconciliation tasks, explicitly confirm authority (production DB vs repo) before editing migrations or docs.
- 2026-02-23: When simplifying migration strategy, separate repo cleanup from live DB history: remove legacy SQL files from active workflow, but keep deployed `d1_migrations` records intact for environment safety.
- 2026-02-23: For production DB changes, use Cloudflare D1 Time Travel bookmarks as the pre-change backup checkpoint and persist the exact restore command alongside the bookmark.
- 2026-02-23: When reporting secret exposure severity, first verify whether the file is actually tracked/pushed (`git ls-files`, `git log -- <path>`) before classifying it as a committed leak.
- 2026-02-23: When tightening webhook security responses, check integration-test contract expectations first; if payload fields are externally consumed or tested, prefer redacting logs over changing response shape.
- 2026-02-25: For compliance fixes that must be reviewed externally (Twilio/registrations), verify the live public site content after deploy (not just local code) before recommending resubmission.
- 2026-03-06: For component tests that depend on both the current date and post-hydration state, control the render month and date helpers separately; fake timers can fix `new Date()` state, but Testing Library polling helpers can stall under fake timers unless assertions stay synchronous.
- 2026-03-07: Failure mode: rendering closed-poll winners from `created_event_id` instead of the poll's persisted `winning_*` ids; detection signal: a closed poll lost its visible winner after the admin chose not to create an event during close; prevention rule: when a workflow stores outcome ids on the source record, member-facing history screens should read from that persisted outcome first and treat derived child records as optional.
- 2026-03-12: Failure mode: sending one outbound email request per recipient sequentially inside a Workers `waitUntil()` task, causing partial fan-out when background/runtime limits cut the job short; detection signal: some members received an event invite while later recipients did not; prevention rule: background fan-out work should use bounded parallelism or a durable queue/outbox rather than a strictly sequential loop.
- 2026-03-12: Failure mode: assuming a production Resend API key can manage domains/webhooks when it is scoped only for send operations; detection signal: live worker logs showed `401 restricted_api_key` on `GET /domains`; prevention rule: verify provider API key scopes before depending on setup automation or admin actions that call management endpoints.
- 2026-03-13: Failure mode: treating Resend webhook management as fire-and-forget even though the API enforces a low per-second request limit; detection signal: live scheduled-worker logs showed `429 rate_limit_exceeded` during webhook creation; prevention rule: provider setup code should honor `Retry-After` and retry rate-limited management requests with backoff.
- 2026-03-13: Failure mode: using the wrong request field name for Resend webhook creation (`url` instead of `endpoint`); detection signal: live scheduled-worker logs showed `422 missing_required_field` with `Missing \`endpoint\` field.`; prevention rule: when wiring provider management APIs, confirm the exact request schema against the current official API docs and add a regression test that asserts the outbound payload shape.
- 2026-03-13: Failure mode: assuming operationally relevant event timing from earlier incident examples instead of re-checking the live latest-event records; detection signal: user corrected that the latest events were not in the past; prevention rule: before proposing resend/backfill steps for production events, verify the current event ids, dates, and status directly from the live database.
- 2026-03-13: Failure mode: mutating an outer variable inside `runInTransaction()` and expecting TypeScript to narrow it afterward; detection signal: `tsc` reported `Property 'recipientCount' does not exist on type 'never'` in the resend branch; prevention rule: when a transaction callback needs to produce a value for later use, return it from `runInTransaction()` instead of assigning through outer scope.
- 2026-03-13: Failure mode: using raw SQL `BEGIN TRANSACTION` / `COMMIT` / `ROLLBACK` with Cloudflare D1 bindings; detection signal: production resend failed before commit, and a direct remote D1 query returned error code `7500` rejecting `BEGIN TRANSACTION`; prevention rule: for D1 worker code, use supported APIs such as `db.batch()`/sessions instead of issuing SQL transaction statements directly.
- 2026-03-13: Failure mode: treating a queue consumer batch as safe to `Promise.all` even when the upstream mail provider enforces a low shared per-second rate limit; detection signal: production resend rows entered `retry` with `Failed to send email: Too Many Requests`, while Resend docs confirmed a 2 requests/second team-wide limit with no burst allowance; prevention rule: when integrating with rate-limited providers, cap worker concurrency explicitly, pace per-request sends inside each batch, and honor provider retry headers instead of retrying on a fixed synchronized schedule.
- 2026-03-13: Failure mode: adding a new route file and assuming React Router type generation will discover it automatically in a repo that uses an explicit route manifest; detection signal: `tsc` failed on missing `./+types/...` for the new admin announcements page until `app/app/routes.ts` and route-health expectations were updated; prevention rule: when adding routes in this app, update the explicit manifest and any route inventory tests before relying on generated route types.
- 2026-03-13: Failure mode: importing a server-only Markdown/email rendering helper into a route component just to reuse preview config; detection signal: the new announcements page briefly depended on a module that also imported `react-dom/server`, which would have polluted the client bundle boundary; prevention rule: keep server-only render helpers isolated, and duplicate or split lightweight preview config when a route component needs client-safe rendering.
- 2026-03-13: Failure mode: reaching for a terminal-side ops workaround when the real need was a normal in-app admin workflow; detection signal: repeated attempts to tunnel live announcement sends through local tooling without browser auth, followed by user correction; prevention rule: when an action is fundamentally an admin product workflow, prefer extending the existing authenticated UI/route path over building side-channel execution tooling.
- 2026-03-13: Failure mode: assuming a reported event-edit error came from the latest pushed admin route without matching the visible UI to a specific route first; detection signal: the screenshot showed the member-facing `Upcoming Events` page, which did not match the clean `main` admin events screen I initially traced; prevention rule: when debugging from screenshots, identify the rendered route/layout from visible copy before choosing which branch or code path to investigate.
- 2026-03-14: Failure mode: leaving an unsupported Cloudflare D1 transaction abstraction available after the first raw-`BEGIN` incident, which let new event workflows reuse the same broken pattern; detection signal: a second user-facing event-edit failure traced back to `runInTransaction()` and older poll routes still issuing raw transaction SQL; prevention rule: after a platform incompatibility is discovered, remove or hard-fail the offending abstraction and audit all call sites immediately instead of patching only the first broken route.
